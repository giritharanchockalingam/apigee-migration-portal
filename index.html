SOLUTION ARCHITECTURE

Today vs. Target

Today: all on MuleSoft. Target: split by workload fit — Apigee for sync, MuleSoft for orchestration.

Current State · MuleSoft-only Integration Platform

As of May 2026
Single Vendor
Every API on the same platform from the same supplier — increased lock-in and licensing leverage.
One-Size-Fits-All
Synchronous proxies, async orchestration, and bulk batch all run on the same runtime — no specialization.
Premium Cost
MuleSoft's pricing model treats every API equally; sync proxies pay the same as complex orchestration.

Future State · Workload-aligned Integration Portfolio

Target · Q4 2026
Why move the sync APIs
The integration team stops running middleware servers, patching runtimes, and sizing CloudHub workers for consumer-facing traffic. Time saved goes back to building features.
Why keep MuleSoft for the rest
Orchestration and batch flows already work. Forcing them onto Apigee just rewrites code that wasn't broken — and Apigee isn't designed for that workload anyway. Don't pay twice.
What "Phase 2" actually means
After the first wave proves out, the 170 async APIs get a second look. Decide then what (if anything) to move later. No commitment up front; the option stays open.

Decision Tree · Apigee X vs MuleSoft Retain vs Phase 2

10 rules · first match wins

Apply each rule in order. The first rule that matches an API determines its platform placement. Any engineer at SEI can walk a new endpoint through this in 30 seconds.

What the tree typically returns
Roughly three out of four APIs fit Apigee. Most of the rest stay on MuleSoft. A handful are genuinely ambiguous and worth waiting on. That's the realistic split — not "100% Apigee" because no estate looks like that.
The team runs it, not the architect
After handover, the integration team applies the tree on every new API. No need to call back to ask "which platform?" — and that's intentional. The decision stays internal.
For when audit asks
Every placement decision gets recorded against the proxy in version control with the rule number that justified it. When compliance or audit asks "why is this here," the answer is one git log away.

Architectural Translation · MuleSoft Hierarchy → Apigee Governance

The discipline migrates, not the pattern

Mule enforces layering at runtime. Apigee doesn't — discipline replaces the framework.

Four architectural options I evaluated

OPTION A
Flatten the hierarchy entirely
Each Apigee proxy talks to backends directly. No layer abstraction.
+ Fewest hops · lowest latency
− Loses abstraction · duplicated logic
OPTION B
Preserve hierarchy with ServiceCallouts
Experience proxy → ServiceCallout → Process proxy → ServiceCallout → System proxy.
+ Mirrors Mule abstraction · familiar
− Three hops · +150–200ms latency · billable internal calls
OPTION C
API Products as the abstraction
One proxy per backend; multiple Products surface it with different policies per consumer type.
+ Single proxy · no extra hops
− Doesn't match Mule source structure · business logic doesn't always fit
OPTION D ★ CHOSEN
Hybrid · mirror Mule projects, flatten internally
Three Apigee proxies match the three MuleSoft projects. Each calls backends directly. ServiceCallout only where genuine multi-source composition exists.
+ Operationally familiar · no inter-proxy hops · uniform FERPA shared flows
− 3-layer abstraction becomes governance-enforced, not runtime-enforced
THE TRADE-OFF
What Mule enforces in the runtime, Apigee enforces in code review. The pattern doesn't migrate — the discipline does.

Mirror Approach vs Native Apigee Architecture

Migration for the 115 · Native for everything next

Mirror for the migration. Native for everything after.

Side-by-side trade-offs

DIMENSION MIRROR (CURRENT POC) NATIVE APIGEE (FORWARD STANDARD)
Proxy decomposition 3 proxies named after Mule layers (system/experience/process). Inherits Mule's structure. 9 proxies named after business domains (courses/enrollments/assignments/...). Inherits the domain model.
Cross-cutting policies OAuth, quota, FERPA inlined per proxy. Three copies. Change costs three edits + three redeploys. Shared Flows referenced by every proxy. One copy. Change costs one edit + one redeploy. Every proxy gets it.
Consumer access Tokens + URLs handed out per-consumer. RBAC ad-hoc. No catalog. API Products bundle proxies for consumers (student-app, faculty-app, sis-integration, analytics). Each Product has its own scopes, quotas, FERPA policy. Developer Portal for self-serve.
Where composition lives Inside Apigee (process-canvas-sis orchestrates in JS policies). Apigee doing Mule's job. Outside Apigee — composition lives in BFFs or domain microservices. Apigee stays a gateway, not a runtime.
Client contract impact Zero. Re-point host. 28 apps unchanged. Breaking. 28 apps must update URLs + OAuth scopes. Coordinated cutover.
Engagement timeline 14 weeks for 115 APIs. 24-30 weeks (adds redesign + client coordination + parallel-run).
Apigee features used ~30%. Proxies, KVMs, basic policies. Apigee acts as a faster MuleSoft. ~90%. API Products, Shared Flows, OAuth2 scopes, monetization hooks, Apigee analytics, Developer Portal.
Best fit for Migrating in-flight APIs where client contracts must be preserved and runway is tight. Greenfield APIs, major refactors, monetized API offerings. Built right from day one.
