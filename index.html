Today vs. Target

Capella's integration estate today runs entirely on MuleSoft. The target state splits workloads by their natural fit: Apigee X handles synchronous APIs with cloud-native governance; MuleSoft retains the orchestration, batch and eventing where it excels.

Architectural Translation · MuleSoft Hierarchy → Apigee Governance

The discipline migrates, not the pattern

MuleSoft enforces the 3-layer API-led architecture at runtime — Experience APIs cannot bypass Process APIs to reach backends. Apigee X has no such concept; every proxy is independent. The migration is not a 1-to-1 port; it is a translation. Below: the side-by-side, the four architectural options I evaluated, and the trade-off the chosen approach makes explicit.

Four architectural options I evaluated

OPTION A
Flatten the hierarchy entirely
Each Apigee proxy talks to backends directly. No layer abstraction.
+ Fewest hops · lowest latency
− Loses abstraction · duplicated logic
OPTION B
Preserve hierarchy with ServiceCallouts
Experience proxy → ServiceCallout → Process proxy → ServiceCallout → System proxy.
+ Mirrors Mule abstraction · familiar
− Three hops · +150–200ms latency · billable internal calls
OPTION C
API Products as the abstraction
One proxy per backend; multiple Products surface it with different policies per consumer type.
+ Single proxy · no extra hops
− Doesn't match Mule source structure · business logic doesn't always fit
OPTION D ★ CHOSEN
Hybrid · mirror Mule projects, flatten internally
Three Apigee proxies match the three MuleSoft projects. Each calls backends directly. ServiceCallout only where genuine multi-source composition exists.
+ Operationally familiar · no inter-proxy hops · uniform FERPA shared flows
− 3-layer abstraction becomes governance-enforced, not runtime-enforced
THE TRADE-OFF MADE EXPLICIT
MuleSoft enforces layer separation at runtime — the framework physically prevents an Experience API from calling a backend directly. Apigee X cannot enforce this; every proxy can call anything. Replacing runtime enforcement with governance enforcement (naming, code review, ADRs, RBAC) is the single largest non-technical deliverable of this engagement. The pattern does not migrate. The discipline does.

MIGRATION APPROACH

Four-wave engagement plan

Two senior engineers · 14 weeks · coaching model · repeatable bundle generator pipeline. The same Node.js script that produced the bundles for this demo runs at Capella against their real Canvas tenant — change one config line, redeploy, done.

Wave 1 · Weeks 4–6
15 simple pass-throughs. Volume builds the team's Apigee muscle memory. No FERPA-sensitive data.
Wave 2 · Weeks 6–9
30 APIs: 20 pass-throughs + 10 request/response. First FERPA-sensitive endpoints land here.
Wave 3 · Weeks 8–11
37 APIs: complex pass-throughs + moderate R/R. Coaching shifts from pairing to code review.
Wave 4 · Weeks 11–14
33 R/R tail including ≥16-score outliers. Advisors focus on Workday SOAP + cutover playbooks.
